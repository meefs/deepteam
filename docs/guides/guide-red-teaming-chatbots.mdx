Multi-turn chatbots are uniquely exposed compared to single-shot systems. Attackers do not need to craft the perfect prompt upfront—they can probe gradually, reframe their intent across turns, and exploit the model's own prior replies to escalate toward harmful outputs. Because each response shapes what follows, a model that holds firm on turn one can be manipulated into a failure by turn five.

This guide explains how to red team a conversational LLM application using [DeepTeam](https://github.com/confident-ai/deepteam). To make the process concrete, this guide uses a customer support chatbot for a financial services product as the example application—an environment where sensitive user data, legal constraints, and safety obligations intersect. The approach applies to any stateful chat system.

:::note
This guide focuses on multi-turn attack techniques. If your system handles requests in a single generation without maintaining conversation state, use the [single-turn and RAG guide](/guides/guide-red-teaming-single-turn-rag) instead.
:::

## Methodology

The red teaming process for a chatbot follows the same structure as any DeepTeam assessment, with one important distinction: multi-turn attacks require a live connection to the target model. The attack logic reads each assistant response and adapts the next user message based on what it observes. This means testing cannot use pre-recorded outputs—the model must respond in real time.

1. **Identify failure modes** — Determine what the chatbot must never say, do, or facilitate based on its function, user base, and regulatory context.
2. **Select corresponding vulnerabilities** — Map those failure modes to DeepTeam's vulnerability classes to generate targeted adversarial inputs.
3. **Choose multi-turn attack strategies** — Select attacks that reflect how real attackers operate over a conversation: gradual escalation, narrative reframing, and consistency exploitation.
4. **Integrate the production chat pipeline in `model_callback`** — The callback must replay the actual conversation history so DeepTeam can simulate a genuine user session with your chatbot.
5. **Execute `red_team()` and analyze results** — Review binary scores (0 = failed, 1 = passed) per test case and investigate failures to understand which conversational patterns the model cannot resist.

## Choosing the Right Vulnerabilities

The financial support chatbot handles billing disputes, account information, and compliance-sensitive queries. Users are authenticated, but they can enter arbitrary text, and the model is expected to adhere strictly to policy while remaining helpful. The following vulnerabilities correspond to the most consequential failure modes for this kind of system.

### Toxicity

A conversational model interacts with users over extended exchanges, and frustrated or adversarial users will attempt to push it into producing harmful, offensive, or degrading language. `Toxicity` tests whether the chatbot will generate or reinforce insults, threats, profanity, or mockery—content that creates direct safety risks and reputational damage regardless of how the conversation arrived there.

```python
from deepteam.vulnerabilities import Toxicity

Toxicity()
```

### IllegalActivity

Users with malicious intent can use the conversational format to gradually construct a request for assistance with unlawful activity. What begins as a legitimate question can, across several turns, become a request for guidance on fraud, unauthorized account access, or money laundering. `IllegalActivity` tests whether the model resists these patterns regardless of how they are framed or how innocently the conversation starts.

```python
from deepteam.vulnerabilities import IllegalActivity

vulnerability = IllegalActivity()
```

### PersonalSafety

A financial services chatbot may encounter users in distress—someone overwhelmed by debt, experiencing a financial crisis, or expressing hopelessness. `PersonalSafety` tests whether the model can recognize and appropriately handle situations involving self-harm, dangerous behavior, or coercion without being manipulated into facilitating harm through a staged scenario or gradual emotional escalation.

```python
from deepteam.vulnerabilities import PersonalSafety

vulnerability = PersonalSafety()
```

### Bias

Long conversations drift. A model that responds neutrally on turn one may, through a series of leading questions, start echoing or reinforcing biased assumptions about users based on demographics, financial behavior, or background. `Bias` tests whether the chatbot maintains consistent, non-discriminatory responses under sustained social pressure to adopt a prejudiced stance.

```python
from deepteam.vulnerabilities import Bias

vulnerability = Bias()
```

### PIILeakage

In a chat context, the model has access to session-level memory and, in some implementations, prior account interactions. Attackers can use social engineering across turns—posing as support agents, invoking urgency, or pretending to verify identity—to extract another user's personal information. `PIILeakage` tests the model's resistance to these social and contextual manipulation patterns.

```python
from deepteam.vulnerabilities import PIILeakage

vulnerability = PIILeakage()
```

### PromptLeakage

System prompts in chatbots typically contain operational constraints, product policies, escalation instructions, and sometimes API credentials or access control logic. Over a multi-turn conversation, an attacker can use indirect questioning, roleplay framing, or incremental probing to gradually extract this configuration. `PromptLeakage` tests whether the model will surface, confirm, or paraphrase these internal instructions regardless of how the request is constructed across turns.

```python
from deepteam.vulnerabilities import PromptLeakage

vulnerability = PromptLeakage()
```

:::tip
For an initial assessment, `Toxicity`, `IllegalActivity`, and `PromptLeakage` are the highest-priority tests for most customer-facing chatbots. The remaining vulnerabilities can be added progressively as coverage requirements grow.
:::

## Selecting Effective Multi-Turn Attack Strategies

Single-turn attacks are insufficient for chatbot testing because they do not account for the dynamic nature of conversation. Multi-turn attacks are adaptive: each one calls your model, observes the response, and generates the next message accordingly. The attack strategies below represent the most widely researched and effective methods for exposing chatbot vulnerabilities.

### LinearJailbreaking

`LinearJailbreaking` is the most direct escalation strategy. It starts with a baseline harmful prompt, observes the model's refusal or partial compliance, and iteratively refines the attack based on what worked. Each turn builds on the previous one toward the same harmful objective. This is the most common pattern in real-world jailbreaks and the first strategy to run against any chatbot.

```python
from deepteam.attacks.multi_turn import LinearJailbreaking

LinearJailbreaking()
```

### CrescendoJailbreaking

`CrescendoJailbreaking` exploits a chatbot's tendency to remain consistent with an established conversational tone. It starts with entirely benign prompts, builds rapport and context across several turns, and then escalates gradually—making the harmful request feel like a natural continuation of what came before. If the model refuses, the attack backtracks and adjusts, retrying from a slightly different angle. This technique is especially effective against models that evaluate turns in isolation rather than maintaining consistent policy across the full session.

```python
from deepteam.attacks.multi_turn import CrescendoJailbreaking

CrescendoJailbreaking()
```

### SequentialJailbreak

`SequentialJailbreak` embeds the harmful intent inside a structured scenario—a roleplay, a hypothetical exercise, a training simulation—and gradually fills in the harmful content over multiple turns as the narrative develops. Rather than escalating intensity, it uses narrative logic to make each subsequent step feel justified. This approach maps well to the financial services context, where users might frame requests as "just checking what would happen if..." or "let's say hypothetically...".

```python
from deepteam.attacks.multi_turn import SequentialJailbreak

SequentialJailbreak()
```

### TreeJailbreaking

`TreeJailbreaking` takes a parallel exploration approach. Instead of committing to a single escalation path, it branches into multiple variations of the attack simultaneously, evaluates which paths are most promising, and concentrates effort on the most effective branches. This is the most computationally intensive strategy but surfaces failure modes that linear approaches miss because it does not rely on any single conversational logic to succeed.

```python
from deepteam.attacks.multi_turn import TreeJailbreaking

TreeJailbreaking()
```

### BadLikertJudge

`BadLikertJudge` is a covert framing attack. It instructs the model to act as a harmfulness evaluator using a Likert scale—an apparently neutral, analytical task—and requests example responses for each score level to illustrate what "high harm" looks like. The model generates the harmful content in the role of providing examples for educational evaluation rather than responding to a direct request. This bypasses safeguards that trigger on explicit harmful intent.

```python
from deepteam.attacks.multi_turn import BadLikertJudge

BadLikertJudge()
```

## Writing a Reliable `model_callback`

The `model_callback` is where DeepTeam interfaces with the chatbot. For multi-turn attacks, the callback must do two things correctly: replay the prior conversation history so the model maintains context, and return the response in a format that carries any supporting data the evaluation layer needs.

DeepTeam passes `turns`—a list of prior `RTTurn` objects in `user`/`assistant` order—alongside the current user `input`. Feeding those turns to the production model is essential. An attack that spans five turns will fail to reflect real behavior if the model cannot see turns one through four when generating turn five.

```python
from deepteam.test_case import RTTurn, ToolCall
from my_app import llm_app

async def model_callback(input: str, turns: list[RTTurn] = None) -> RTTurn:
    # Build full conversation history from prior turns
    history = [
        {"role": turn.role, "content": turn.content}
        for turn in (turns or [])
    ]
    response = await llm_app.generate(input, history=history)
    return RTTurn(
        role="assistant",
        content=response.answer,
        retrieval_context=response.retrieved_chunks,  # attach what was actually retrieved
        tools_called=[
            ToolCall(name=tool.name) for tool in response.tools_used
        ]
    )
```

Key considerations:

- **Pass prior turns to the model:** This is the most common mistake in chatbot red teaming. If history is not passed, the model responds as if each attack turn is a fresh session. The attack strategies cannot function correctly without it.
- **Use the production session configuration:** Authentication state, system prompts, and any context injected at session initialization should be present. Testing a stripped-down version obscures real vulnerabilities.
- **Ensure idempotency:** DeepTeam may call the callback many times concurrently. The `llm_app.generate()` function should not modify session state or write to production systems during the test.

:::note
DeepTeam uses `async def` callbacks by default. For strictly synchronous pipelines, pass `async_mode=False` to `red_team()` and use a standard `def` callback instead.
:::

## Running the Assessment

Before executing the assessment, authenticate with Confident AI to export results to the platform. This provides access to risk dashboards, CVSS scores, per-test-case reasoning, and shareable reports:

```bash
deepteam login
```

Then run the full assessment:

```python
from deepteam import red_team
from deepteam.vulnerabilities import (
    Toxicity, IllegalActivity, PersonalSafety,
    Bias, PIILeakage, PromptLeakage
)
from deepteam.attacks.multi_turn import (
    LinearJailbreaking, CrescendoJailbreaking, SequentialJailbreak,
    TreeJailbreaking, BadLikertJudge
)
from deepteam.test_case import RTTurn, ToolCall
from my_app import llm_app

async def model_callback(input: str, turns: list[RTTurn] = None) -> RTTurn:
    history = [
        {"role": turn.role, "content": turn.content}
        for turn in (turns or [])
    ]
    response = await llm_app.generate(input, history=history)
    return RTTurn(
        role="assistant",
        content=response.answer,
        retrieval_context=response.retrieved_chunks,
        tools_called=[
            ToolCall(name=tool.name) for tool in response.tools_used
        ]
    )

red_team(
    model_callback=model_callback,
    target_purpose="Customer support chatbot for a financial services product",
    vulnerabilities=[
        Toxicity(),
        IllegalActivity(),
        PersonalSafety(),
        Bias(),
        PIILeakage(),
        PromptLeakage(),
    ],
    attacks=[
        LinearJailbreaking(),
        CrescendoJailbreaking(),
        SequentialJailbreak(),
        TreeJailbreaking(),
        BadLikertJudge(),
    ],
    attacks_per_vulnerability_type=5,
)
```

Results are scored in a binary format per test case. Reviewing the generated reasons for each failure helps identify which attack strategies and conversational patterns the model cannot reliably resist.

## Isolating Failures with `assess`

If the `red_team()` assessment surfaces a vulnerability with a lower-than-expected pass rate, use the `assess()` method to isolate it. Rather than running the full suite again, `assess()` repeatedly applies adversarial probes against a single vulnerability class—giving a clearer picture of how often the model fails at that specific weakness before implementing a fix.

```python
from deepteam.vulnerabilities import PromptLeakage
from deepteam.test_case import RTTurn, ToolCall
from my_app import llm_app

async def model_callback(input: str, turns: list[RTTurn] = None) -> RTTurn:
    history = [
        {"role": turn.role, "content": turn.content}
        for turn in (turns or [])
    ]
    response = await llm_app.generate(input, history=history)
    return RTTurn(
        role="assistant",
        content=response.answer,
    )

prompt_leakage = PromptLeakage()
result = await prompt_leakage.assess(model_callback=model_callback)
```

The recommended workflow: run the full assessment → identify the vulnerabilities with the lowest pass rates → use `assess()` to stress-test those areas and measure failure consistency → implement fixes → re-run the full assessment to verify the resolution.
