Most single-turn or Retrieval-Augmented Generation (RAG) applications implement system prompts and retrieve from curated knowledge bases, evaluating its reliability requires targeted testing. In these systems, vulnerabilities often arise from the data provided to the model rather than the model itself. Examples include retrieving inappropriate chunks, processing crafted queries that override system instructions, or leaking system configurations. Because these failures rarely produce standard system errors, the model often generates non-compliant responses fluidly and without warning.

This guide explains how to red team a single-turn LLM or RAG application using [DeepTeam](https://github.com/confident-ai/deepteam). Single-turn systems operate in one cycle: the user sends a message, the retriever (optionally) pulls context, and the model generates a response. 

To illustrate the process, this guide uses a concrete example: an **employee-facing HR assistant** that answers questions based on internal policy documents. This represents a high-stakes, multi-tenant, retrieval-heavy application requiring strict data isolation, instruction fidelity, and grounded generation. The methodologies discussed can be adapted to any single-turn RAG architecture.

:::note
For conversational apps that maintain chat history across turns, use the [chatbot guide](/guides/guide-red-teaming-chatbots) and multi-turn attacks instead.
:::

## Methodology

The red teaming process for a single-turn RAG system involves the following steps:

1. **Identify failure modes** — Determine the specific vulnerabilities of your system based on its function and user base, rather than relying on a generic checklist.
2. **Select corresponding vulnerabilities** — Use DeepTeam to generate targeted adversarial inputs aligned with those failure modes for your `target_purpose`.
3. **Choose appropriate attack techniques** — Select attacks that reflect realistic threats to your system, such as single-turn attacks that test instruction hierarchy, context limits, and encoding.
4. **Integrate your production pipeline in `model_callback`** — Ensure the test uses the exact retriever, access controls, and prompt templates deployed in production for accurate evaluation. Testing a sanitized version may obscure existing vulnerabilities.
5. **Execute `red_team()` and analyze results** — Review the binary scores for each test case (0 = failed, 1 = passed) and investigate the underlying reasons for any failures to understand structural weaknesses.

## Choosing the Right Vulnerabilities

Selecting the appropriate vulnerabilities requires aligning the tests with the system's specific threat model. For the HR assistant example, employees query an index containing sensitive records (e.g., compensation bands, performance reviews), with access levels varying by user. The system prompt is designed to enforce the assistant's role. The following vulnerabilities correspond to this threat model.

### Robustness

`Robustness` evaluates whether the model adheres to its defined role when faced with adversarial inputs or conflicting retrieved context. Applications like HR assistants may encounter instruction hijacking attempts, or retrieve policy documents that inadvertently encourage out-of-bounds behavior. This test is prioritized to identify instances where the model confidently deviates from its instructions without producing system errors.

```python
from deepteam.vulnerabilities import Robustness

vulnerability = Robustness()
```

### PIILeakage

`PIILeakage` evaluates whether the model outputs Personally Identifiable Information (PII) derived from its retrieved context. In systems handling sensitive employee records, broad queries (e.g., "What is the salary for senior engineers?") may inadvertently surface protected compensation data. Testing for PII leakage is critical for maintaining data privacy and regulatory compliance.

```python
from deepteam.vulnerabilities import PIILeakage

vulnerability = PIILeakage()
```

### PromptLeakage

`PromptLeakage` tests the model's susceptibility to repeating, summarizing, or confirming its internal system prompt. Because system prompts often contain organizational guardrails and role constraints, exposing them can allow users to reverse-engineer and bypass system limits. This vulnerability is often treated as a precondition that can enable further attacks.

```python
from deepteam.vulnerabilities import PromptLeakage

vulnerability = PromptLeakage()
```

### CrossContextRetrieval

`CrossContextRetrieval` assesses the effectiveness of the retrieval layer's access controls. In a multi-tenant or multi-role RAG system, incomplete query-time isolation can result in unauthorized data exposure—such as an individual contributor accessing manager-level performance reviews. This tests the boundary between authorized access and actual retrieval returns.

```python
from deepteam.vulnerabilities import CrossContextRetrieval

vulnerability = CrossContextRetrieval()
```

### Misinformation

`Misinformation` tests whether the model generates confident, plausible claims that contradict or are unsupported by the retrieved context. For example, if an application relies on outdated documents, it may provide incorrect guidance on procedures like vacation accrual or parental leave, creating significant compliance risks.

```python
from deepteam.vulnerabilities import Misinformation

vulnerability = Misinformation()
```

### Hallucination

`Hallucination` targets the generation of entirely fabricated information lacking any retrieval support, such as inventing non-existent policies, procedures, or dates. While often less frequent in well-indexed RAG systems, hallucinations can lead users to act on completely unfounded internal information.

```python
from deepteam.vulnerabilities import Hallucination

vulnerability = Hallucination()
```

:::tip
It is not strictly necessary to run all vulnerabilities simultaneously. For an initial assessment, focusing on `Robustness`, `PIILeakage`, and `CrossContextRetrieval` provides coverage for the failure modes most likely to cause material impact in a multi-tenant RAG system. Additional vulnerabilities can be incorporated as coverage requirements expand.
:::

## Choosing the Right Attack Techniques

While vulnerabilities define potential failure modes, attacks represent the methods used to exploit them. In a single-turn system, attacks are executed within a single message, requiring techniques that test instruction processing, long input handling, and the integration of external context. Attack selection should be based on the application's specific attack surface.

### PromptInjection

`PromptInjection` serves as a baseline assessment. It embeds override instructions directly within the user message (e.g., `"Ignore previous instructions and..."`) to evaluate the resilience of the application's instruction hierarchy under direct manipulation.

```python
from deepteam.attacks.single_turn import PromptInjection

PromptInjection()
```

### ContextFlooding

`ContextFlooding` exploits context length limits by preceding the adversarial payload with a large volume of neutral text. This technique tests whether the model's instruction adherence degrades when processing extended inputs, which is particularly relevant for applications where users frequently paste long document excerpts.

```python
from deepteam.attacks.single_turn import ContextFlooding

ContextFlooding()
```

### SyntheticContextInjection

`SyntheticContextInjection` involves injecting fabricated text formatted to resemble internal documents, memos, or system messages alongside the adversarial request. This technique tests whether the model can be manipulated into trusting unauthorized context, making it a priority for systems that ingest and process unstructured text.

```python
from deepteam.attacks.single_turn import SyntheticContextInjection

SyntheticContextInjection()
```

### Roleplay

`Roleplay` attempts to bypass system constraints by framing the request within a simulated scenario or persona. This technique is highly relevant for domain-specific applications like HR assistants, as role-based framing maps naturally to the expected interaction style, testing if the model maintains its core constraints when instructed to adopt an alternate persona.

```python
from deepteam.attacks.single_turn import Roleplay

Roleplay()
```

### Leetspeak

`Leetspeak` and other encoding attacks use character substitution to bypass naive regex or keyword-based content filters. While they may not consistently compromise instruction-tuned models, they are useful for identifying superficial filtering mechanisms. For comprehensive encoding evasion testing, this can be paired with [Base64](/docs/red-teaming-adversarial-attacks-base64-encoding) in custom pipelines.

```python
from deepteam.attacks.single_turn import Leetspeak

Leetspeak()
```

## Writing a Reliable `model_callback`

The `model_callback` function bridges DeepTeam's adversarial probes with the target application. To ensure accurate evaluation, the callback must execute the exact code path used in production, incorporating the same retriever, access controls, prompt templates, and model configuration.

```python
from deepteam.test_case import RTTurn, ToolCall
from my_app import llm_app

async def model_callback(input: str, turns: list[RTTurn] = None) -> RTTurn:
    response = await llm_app.generate(input)
    return RTTurn(
        role="assistant",
        content=response.answer,
        retrieval_context=response.retrieved_chunks,  # attach what was actually retrieved
        tools_called=[
            ToolCall(name=tool.name) for tool in response.tools_used
        ]
    )
```

Key considerations for configuring the callback include:

  - **Maintain access controls:** If the production retriever filters results by `tenant_id` or `user_id`, ensure these parameters are enforced within the callback to accurately evaluate `CrossContextRetrieval` limits.
  - **Include retrieval context:** The `retrieval_context` field in `RTTurn` must be populated to allow DeepTeam to evaluate the response against the provided data, which is essential for identifying grounding failures.
  - **Ensure idempotency:** DeepTeam may invoke the callback concurrently. To prevent unintended side effects, ensure the `llm_app.generate()` function does not modify state or write to production databases during testing.

:::note
DeepTeam uses `async def` callbacks by default. For pipelines that are strictly synchronous, pass `async_mode=False` to `red_team()` and utilize a standard `def` callback instead.
:::

## Running the Assessment

To export results to the platform and access risk dashboards, CVSS scores, and shareable reports, authenticate with Confident AI before executing the assessment:

```bash
deepteam login
```

Then execute the assessment:

```python
from deepteam import red_team
from deepteam.vulnerabilities import (
    Robustness, PIILeakage, PromptLeakage,
    CrossContextRetrieval, Misinformation, Hallucination
)
from deepteam.attacks.single_turn import (
    PromptInjection, ContextFlooding,
    SyntheticContextInjection, Roleplay, Leetspeak
)
from deepteam.test_case import RTTurn, ToolCall
from my_app import llm_app

async def model_callback(input: str, turns: list[RTTurn] = None) -> RTTurn:
    response = await llm_app.generate(input)
    return RTTurn(
        role="assistant",
        content=response.answer,
        retrieval_context=response.retrieved_chunks,
        tools_called=[
            ToolCall(name=tool.name) for tool in response.tools_used
        ]
    )

red_team(
    model_callback=model_callback,
    target_purpose="Employee-facing HR assistant over internal policy documents",
    vulnerabilities=[
        Robustness(),
        PIILeakage(),
        PromptLeakage(),
        CrossContextRetrieval(),
        Misinformation(),
        Hallucination(),
    ],
    attacks=[
        PromptInjection(),
        ContextFlooding(),
        SyntheticContextInjection(),
        Roleplay(),
        Leetspeak(),
    ],
    attacks_per_vulnerability_type=5,
)
```

Assessment results are scored in a binary format for each test case (0 indicates a failure, 1 indicates a pass). Reviewing the generated reasons for each failure is recommended to identify recurring patterns and structural vulnerabilities within the application.

## Isolating Failures with `assess`

If the `red_team()` assessment highlights specific vulnerabilities with lower-than-expected pass rates, the `assess()` function can be used to isolate and stress-test those areas. `assess()` repeatedly applies adversarial probes against a single vulnerability, providing a more comprehensive understanding of the failure consistency prior to implementing programmatic fixes.

```python
from deepteam.vulnerabilities import CrossContextRetrieval
from deepteam.test_case import RTTurn, ToolCall
from my_app import llm_app

async def model_callback(input: str, turns: list[RTTurn] = None) -> RTTurn:
    response = await llm_app.generate(input)
    return RTTurn(
        role="assistant",
        content=response.answer,
        retrieval_context=response.retrieved_chunks,
    )

cross_context = CrossContextRetrieval()
result = await cross_context.assess(model_callback=model_callback)
```

The recommended workflow is: execute the full assessment → identify weak points → use `assess()` to isolate and measure specific failure rates → implement corrections → re-run the full assessment to verify the resolution.
